# HW-32: Аналіз ANI та архітектура інтеграції ШІ

**Обраний кейс:** Б — Автоматичне сортування вхідних звернень у службу підтримки банку  
(скарги, подяки, технічні питання)

---

## Частина 1. Аналіз та класифікація (теорія)

### 1.1. Чому цей кейс належить до ANI, а не до AGI?

**ANI (Artificial Narrow Intelligence)** — система, навчена вирішувати *одну вузьку задачу* у фіксованому домені.

Кейс Б — класичний ANI, бо:

| Ознака ANI | Як проявляється в кейсі |
|---|---|
| Одна задача | Класифікація тексту звернення в категорії (скарга / подяка / тех. питання) |
| Фіксований домен | Банківська підтримка, обмежений словник і типи звернень |
| Немає загального міркування | Модель не «розуміє» банк загалом і не вміє вести переговори, торгувати чи писати код |
| Метрика якості конкретна | Accuracy / F1 по класах, а не «інтелект загалом» |

**AGI** передбачала б здатність переносити інтелект між довільними задачами (як людина): сьогодні класифікувати звернення, завтра керувати авто, післязавтра писати наукову статтю — без спеціального перенавчання під кожну задачу.  
Сортувальник звернень цього не вміє: поза своєю таксономією він «сліпий».

> Порада з завдання: не робимо «все для всього» — вирішуємо *одну* проблему (маршрутизація звернень) максимально ефективно.

---

### 1.2. Класичне ML чи Deep Learning?

**Рекомендація: класичне ML на табличних/текстових ознаках як baseline; DL — якщо обсяг і якість даних це виправдовують.**

| Підхід | Коли доречний | Приклад |
|---|---|---|
| **Класичне ML** (TF-IDF / bag-of-words + Logistic Regression / SVM / Gradient Boosting) | Мало даних (тисячі прикладів), потрібна швидкість, інтерпретованість, низька вартість | TF-IDF → LogisticRegression |
| **Deep Learning** (BERT / multilingual transformers) | Багато розмічених звернень, складні формулювання, сленг, змішані мови (укр/рос/англ) | Fine-tuned `ukr-roberta` / `multilingual-e5` |

**Обґрунтування для банку:**

1. Звернення — це **текст**, але задача — **класифікація з фіксованими мітками**, не генерація.
2. Для старту достатньо класичного пайплайна: швидкий інференс, зрозумілі фічі, легко дебажити помилки («чому пішло в "скаргу"»).
3. DL варто підключати, коли з’являться десятки тисяч розмічених прикладів і класичне ML «застигне» на ~85–90% accuracy.
4. Повноцінний LLM (GPT) тут **надлишковий і дорогий** для production-сортування; його можна лишити для *допоміжного* підсумку звернення, а не для основної класифікації.

**Вердикт:** старт — **класичне ML**; еволюція — **легкий transformer (DL)**, якщо метрики вимагають.

### 1.3. Який «перенос знань» можливий?

**Узкоспеціалізована модель майже не переноситься «як є».**

| Сценарій | Чи спрацює? | Чому |
|---|---|---|
| Модель з кейсу Б → сортування звернень у *іншому банку* | Частково | Схожий домен, але інші продукти, сленг, шаблони — потрібен fine-tune / retrain |
| Модель з кейсу Б → сортування тікетів IT-підтримки магазину | Слабко | Інший словник («повернення товару» ≠ «блокування картки») |
| Модель з кейсу В (вібрація обладнання) → поломка автомобіля | **Ні без донавчання** | Інші сенсори, частоти, режими роботи, типи відмов — ознаки не збігаються |
| Embedding-модель (загальна мовна) → будь-який текст | Так, як *feature extractor* | Переноситься представлення мови, але **голова класифікатора** тренується заново |

**Висновок:** у ANI «перенос знань» обмежений *близьким доменом* і часто зводиться до reuse embeddings / architecture, а не готової моделі. Модель кейсу В **не передбачить** поломку авто без нових даних і адаптації.

## Частина 2. Архітектурна вправа (практика)

**Контекст інтеграції:** розважальний Telegram-бот «AI-Співрозмовник» (Частина 4) — модуль ШІ для чат-проєкту з характером, пам’яттю діалогу та стилізацією.

### Високорівневий Workflow

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────┐     ┌──────────────┐
│ Data Source │ ──► │    Processing    │ ──► │     Action      │ ──► │    Metric    │
└─────────────┘     └──────────────────┘     └─────────────────┘     └──────────────┘
```

### 2.1. Data Source — звідки дані?

| Джерело | Що саме |
|---|---|
| **Ввід користувача (Telegram)** | Текст повідомлень, команди (`/start`, `/style`, `/rpg`, `/clear`) |
| **Пам’ять діалогу** | Історія останніх N реплік у Redis / in-memory (контекст розмови) |
| **Конфіг / System Prompt** | Personality бота (саркастичний бармен із 2115 року) |
| **(.env)** | API keys, модель, ліміти токенів — не з БД користувачів |

Для *банківського* кейсу Б джерелами були б: CRM/helpdesk БД, email/чат-логи, метадані звернення (канал, продукт).

### 2.2. Processing — де обробка?

| Варіант | Роль у проєкті |
|---|---|
| **Зовнішнє API (OpenAI)** | Основний шлях: `chat.completions` для реплік, стилізації, RPG |
| **Локально (Ollama)** | Опційний fallback для офлайн / економії; той самий system prompt |
| **Локальний шар (Python)** | Збірка контексту, обрізка історії, rate-limit, вибір режиму |

**Обрано для hw_32:** OpenAI API + локальний orchestration у `bot/` (aiogram).  
Причина: швидка інтеграція streaming/context, без розгортання GPU.

### 2.3. Action — що робить система з результатом?

1. Відправляє текстову відповідь у Telegram-чат.
2. Зберігає пару user/assistant у пам’яті діалогу (щоб «пам’ятав» попередні 2 хвилини / N повідомлень).
3. У режимах `/style` і `/rpg` змінює system prompt і поводиться інакше.
4. `/clear` скидає контекст.

*(У банківському кейсі Action був би: запис категорії в тикет-систему, маршрутизація до черги оператора, опційний Slack/email alert.)*

### 2.4. Metric — як зрозуміти якість?

| Метрика | Ціль | Як міряти |
|---|---|---|
| **Latency** | p95 відповіді < 3 с | Лог часу від повідомлення до reply |
| **Контекст** | Бот згадує факти з історії ≥ 80% тест-діалогів | Ручний checklist / eval-сет із 20 діалогів |
| **Personality adherence** | Відповіді відповідають характеру ≥ 4/5 за rubrics | Оцінка людиною / LLM-as-judge |
| **Стабільність** | Error rate API < 2% | Логи винятків, retries |

Для кейсу Б (банк): accuracy класифікації **> 85%**, F1 по мінорних класах, зменшення середнього часу першої маршрутизації на **~30%**.

### Схема інтеграції (бот)

```
Telegram User
     │
     ▼
┌──────────────┐
│  aiogram Bot │  ← webhook / long polling
└──────┬───────┘
       │ 1. message + chat_id
       ▼
┌──────────────┐
│ Memory Store │  ← get/append history (Redis або dict)
└──────┬───────┘
       │ 2. messages[] = system + history + user
       ▼
┌──────────────┐
│ OpenAI API   │  ← chat.completions (або Ollama)
└──────┬───────┘
       │ 3. assistant text
       ▼
┌──────────────┐
│ Telegram reply│ + save to Memory
└──────────────┘
```

## Частина 3. Вузькі місця (Engineering Mindset)

Три ключові ризики для обраного **кейсу Б** (і частково для бота з Частини 4):

### Ризик 1. Конфіденційність даних (PII)
Звернення містять ПІБ, номери карток, ІПН. Відправка сирого тексту в зовнішній LLM-API = витік персональних даних і порушення політик банку / GDPR-аналогів.  
**Мітигація:** класифікація on-prem / VPC, маскування PII перед інференсом, заборона логувати тіло звернення.

### Ризик 2. Drift і помилкова маршрутизація
Нові продукти банку, сезонні кампанії, зміна шаблонів клієнтів → падіння accuracy. Хибна «скарга» замість «тех. питання» → неправильна черга, SLA, репутаційні втрати.  
**Мітигація:** мониторинг confidence + human-in-the-loop при score < порогу; регулярний retrain на свіжих мітках.

### Ризик 3. Latency і вартість (якщо піти в LLM замість класифікатора)
На піку (тисячі звернень/годину) виклик GPT на кожне повідомлення дає затримку і великий рахунок.  
**Мітигація:** легкий класифікатор (ML/BERT) для сортування; LLM — лише для складних кейсів або резюме.

**Додатково для розважального бота (Ч.4):** галюцинації (вигадані факти «з 2115»), витік API-ключа з `.env`, rate limits Telegram/OpenAI — див. README проєкту.

## Зв’язок із Частиною 4

Практична реалізація ANI-підходу: **один фокус** — розважальний співрозмовник із фіксованою personality, контекстом і режимами стилізації/RPG.  
Не намагаємось будувати AGI-асистента «для всього світу» — лише якісний чат із характером і керованим контекстом.
